## Context

Block 1 built the execution engine. Block 2 built the contract layer that turns a directory of prompts into a runnable agent. Career-ops is the first real consumer — the one that proves the contract is correct or exposes where it's wrong.

The career-ops domain agent gives Roberto job search capabilities: evaluate postings against Daniel's profile, triage pipelines, research companies, draft outreach, and track applications. All intelligence lives in prompt files, not Python. The runtime provides the gates (trust, budget, lifecycle), the registry provides the wiring (manifest → checklist → runtime), and this step provides the domain knowledge.

Two open-source implementations exist. Andrew Shwetzer's Cowork adaptation is the primary source — it's already prompt-first with 9 skills, reference docs, and a profile schema. Santiago Ferreira's original is reference-only for evaluation rubric depth and archetype design, but its Node.js/Go/Playwright infrastructure is irrelevant to Xibi.

**What this step validates:** Can a third party's prompt-based plugin be ported to Xibi's domain agent contract with zero core code changes? If this works, tourism and meeting-prep follow the same pattern.

```
    skills/

        triage.md                   # Quick-score pipeline (L1, haiku)
        track.md                    # Application tracking (L1, haiku)
        apply.md                    # Application form filling (L2, sonnet)
        research.md                 # Company intelligence brief (L1, sonnet)
        outreach.md                 # LinkedIn/email message drafts (L2, opus)
        compare.md                  # Side-by-side opportunity comparison (L1, sonnet)
    references/
        archetypes.md               # 15 industry archetypes + detection algorithm
        scoring-rubric.md           # 10 weighted dimensions, persona modifiers
        ats-rules.md                # ATS optimization rules for resume tailoring
        profile-schema.md           # Documents every field in profile.yml
        states.md                   # Canonical application statuses
    output/                         # Presentation files land here (per step-82 convention)
        {run_id}.md                 # Generated by SummaryGenerator
        story-bank.md               # Accumulated STAR+R stories across evaluate runs (persistent)
```

### What Is NOT Built (deferred)

- **`scan` skill** — Requires HTTP fetch capability that the subagent runtime doesn't have. Portal scanning hits Greenhouse/Ashby/Lever JSON APIs (public, no auth). This needs either a `tools: ["http_fetch"]` declaration in the manifest with a corresponding trust-gated runtime tool, or an MCP server wrapping those endpoints. Either way, it's more than this step should bite off. Scan is deferred to a follow-up step.
- **`default_sequence`** — Without scan, the full pipeline (scan → triage → evaluate) can't run. Each skill ships as `standalone: true` for this step. When scan is added, default_sequence gets defined.
- **PDF generation** — tailor-resume outputs markdown. PDF rendering is a future enhancement.
- **Batch processing** — Parallel evaluation of multiple postings. Requires runtime support for fan-out, which is epic backlog.

## Skill Prompt Design

### Reference Doc Loading (critical runtime constraint)

The LLM executing a checklist step has **no file tools** — it receives a prompt string and nothing else. Skill prompts that say "Read references/archetypes.md" won't work. Instead, Block 2's `resolve_checklist` handles this at checklist-build time:

1. When resolving a skill, scan the prompt file for `references/*.md` mentions
2. Load each referenced file's content
3. Inject the content into `scoped_input.references` as a dict: `{"archetypes.md": "<content>", "scoring-rubric.md": "<content>"}`
4. The skill prompt's References section becomes: "The following reference documents are available in your scoped_input.references field:" (not "Read and apply these files")

This means skill prompts must be written for the **injected-content model**, not the tool-access model. Andrew-shwetzer's skills say `Read references/scoring-rubric.md` — every Xibi adaptation must replace those with references to `scoped_input.references["scoring-rubric.md"]`.

Similarly, `story-bank.md` (if it exists) is loaded into scoped_input at resolve time so evaluate can check for existing stories.

### JSON Output Requirement

`checklist.py` (lines 206-215) parses every step's LLM response as JSON. The fallback (`{"raw_content": ..., "error": ...}`) works but produces unstructured blobs that break context chaining, summary generation, and retrieval. **Every skill prompt must end with an explicit JSON output schema** and the instruction "Return ONLY a JSON object, no markdown fences, no commentary outside the JSON."

### Prompt Template

Each skill prompt file follows a consistent structure adapted for Xibi's runtime:

```markdown
# {Skill Name}

## Context
You are a career strategist working as part of Xibi, an autonomous agent framework.
Your output will be stored in a database and summarized for the user (Daniel) by Roberto,
his AI assistant. Write for structured consumption, not conversational chat.

## Input
Your scoped_input contains:
- `user_config.profile` — Daniel's career profile (work history, skills, targets, narrative)
- `references` — dict of reference document contents (keys are filenames)
- {skill-specific inputs described here}

## Reference Documents
The following documents are available in scoped_input.references:
- `archetypes.md` — industry archetypes and detection algorithm
- `scoring-rubric.md` — weighted scoring dimensions
- {skill-specific references}

## Instructions
{Detailed step-by-step rubric — adapted from andrew-shwetzer, deepened from santifer}

## Output Format
Return ONLY a JSON object with this structure (no markdown fences, no text outside the JSON):
{skill-specific output schema with field descriptions}
```

### Skill-Specific Adaptations

**evaluate.md** — The heaviest skill. Adapts the Cowork plugin's 11-step evaluate flow but deepens Block B (Background Match) and Block F (Interview Prep) using santifer's richer STAR+Reflection structure. Key sections: archetype detection (15 archetypes from archetypes.md), **A-G scoring blocks** (Block G = Posting Legitimacy, adapted from santifer), persona modifier application, final 1.0-5.0 score with honest assessment. Model: sonnet (needs judgment quality).

> **Block G — Posting Legitimacy** (from santifer `modes/oferta.md`): Analyzes posting freshness, description quality (specificity vs boilerplate, internal contradictions, realistic requirements), company hiring signals (layoffs/freeze context), reposting detection, and role market context. Three-tier assessment: High Confidence / Proceed with Caution / Suspicious. Edge cases handled: government/academic (60-90 day norms), evergreen/pipeline roles, niche/executive (long fill times), startup (legitimately vague JDs), no-date-available (defaults to caution, never suspicious without evidence), recruiter-sourced (active outreach is itself a positive signal). In v1 without web search, the LLM assesses description quality and internal consistency from text alone — web-enriched signals (Glassdoor, layoff news) activate when the runtime gains tool support.

> **Story Bank Persistence** (from santifer Block F): After generating STAR+R stories, evaluate outputs a `story_bank_updates` array. The summary generator (or a post-run hook) appends new stories to `domains/career-ops/output/story-bank.md`, building a reusable bank across evaluations. On subsequent runs, the story bank file is loaded into scoped_input so evaluate can check for duplicates and reference existing stories instead of regenerating.

**tailor-resume.md** — Reads profile + prior evaluation output (if available). Generates a markdown resume optimized for ATS keyword matching per ats-rules.md. Does NOT generate PDF — outputs structured markdown that Roberto can present. Key: the resume must reflect only real experience from the profile, never fabricated. L2 because it represents the user to employers.

**triage.md** — Lightweight scoring. Takes an array of postings, scores each 1.0-5.0 with a one-line verdict. No deep analysis — that's evaluate's job. Uses scoring-rubric.md but applies only the top 3 weighted dimensions for speed. Model: haiku (fast, cheap). **Input model without scan:** Each posting in the array is `{text: string}` or `{title: string, company: string, text: string}` or `{url: string}` — the user pastes or provides them directly. When scan ships, it will produce this same shape automatically.

**track.md** — Application lifecycle management. Reads/writes to the run's output (which persists via output_ttl_hours). Supports add, update, status query, and pipeline stats. Uses states.md for canonical status values. Model: haiku.

**apply.md** — Prepares application materials: cover letter, form field answers, screening question responses. Reads evaluation output to reuse match analysis. L2 because the output goes to employers. Model: sonnet. **Note:** Santifer has a `disable-model-invocation` flag that prevents the agent from actually submitting applications. In Xibi, L2 trust handles this — apply's output is parked in `pending_l2_actions`, never auto-executed. No equivalent flag needed.

**research.md** — Company intelligence brief. The LLM uses its training knowledge (no web search in v1) to produce: company overview, culture signals, tech stack, funding/financials, leadership, growth trajectory, Glassdoor sentiment summary, and red/green flags. Model: sonnet.

**outreach.md** — Drafts contact messages. Two templates: LinkedIn connection request (300 char limit) and email (3-paragraph structure). Reads evaluation + research context to personalize. L2 because messages go to real people. Model: opus (outreach quality is make-or-break).

**compare.md** — Takes 2-5 evaluated postings and produces a weighted comparison matrix. Dimensions from scoring-rubric.md. Outputs a ranked recommendation with tradeoff analysis. Model: sonnet.

## Reference Documents

These are knowledge files that skill prompts reference. They are NOT executed — they're read by the LLM as context.

### `references/archetypes.md`

Adapted from andrew-shwetzer's `references/archetypes.md`. 15 industry archetypes with keyword lists and scoring weight adjustments:

Technology, Finance, Healthcare, Legal, Creative/Marketing, Operations, Sales/BD, Education, Executive, Trades, Customer Success, People/HR, Government/Nonprofit, Scientific/R&D, Non-Software Engineering.

Each archetype defines: detection keywords (weighted: title 3x, requirements 2x, description 1x), scoring weight overrides, industry-specific evaluation language, and credential requirements (for regulated industries).

**AI sub-archetype merge (from santifer):** Santifer defines 6 AI-specific archetypes (FDE, Solution Architect, PM, LLMOps, Agentic, Transformation) with tailored proof-point priorities and STAR framing per sub-type. These are more granular than the general "Technology" archetype. The implementation adds them as specializations within the Technology archetype: when the primary archetype is Technology, a secondary detection pass runs against the 6 AI sub-archetypes to select tailored evaluation framing (e.g., FDE → prioritize delivery speed and client-facing proof points; Agentic → prioritize orchestration, HITL, error handling).

### `references/scoring-rubric.md`

Adapted from andrew-shwetzer's `references/scoring-rubric.md`. 10 weighted dimensions:

1. Role-skill alignment (weight varies by archetype)
2. Seniority match
3. Industry/domain fit
4. Technical requirements coverage
5. Cultural alignment signals
6. Compensation alignment
7. Growth/trajectory fit
8. Location/remote policy match
9. Company stage match
10. Career narrative coherence

Plus persona modifiers: recent_graduate, career_changer, career_returner, international — each adjusts dimension weights.

Score bands: 4.5-5.0 (Excellent), 3.5-4.4 (Good), 3.0-3.4 (Worth Considering), 2.0-2.9 (Weak), 1.0-1.9 (Poor). Below 3.0 the system explicitly recommends against applying.

### `references/ats-rules.md`

ATS optimization rules for resume tailoring. Keyword injection strategy, formatting constraints (no tables, no columns, no graphics), section ordering, and the "mirror the JD language" principle.

### `references/profile-schema.md`

Documents every field in `config/profile.yml`. This is the reference for both the user (filling out their profile) and the skill prompts (knowing what data is available). Schema covers: identity, current state, target roles, skills, credentials, narrative, work history, education, compensation, proof points, portfolio, and persona modifiers.

### `references/states.md`

Canonical application statuses: Evaluated, Applied, Responded, Interview, Offer, Rejected, Discarded, SKIP. Track skill enforces these — no freeform status values.

## Trigger Integration

Roberto needs to recognize career-ops intents and dispatch to the right skill. This wires into bregger_core.py's KeywordRouter:

```python
# Career-ops trigger patterns (added to KeywordRouter)
# These map user messages to agent_id + skill + scoped_input

# evaluate — user pastes a job posting or URL
# Pattern: "evaluate this", "should I apply", "rate this", "how well do I match"
# Also: any message containing what looks like a job description (heuristic: >200 chars with role/company/requirements keywords)
# → agent_id="career-ops", skills=["evaluate"], scoped_input={"posting": <parsed content>}

# tailor-resume — user asks for a resume
# Pattern: "tailor my resume", "make a resume for", "ATS resume"
# → agent_id="career-ops", skills=["tailor-resume"], scoped_input={"posting": <context>}

# research — user asks about a company
# Pattern: "research {company}", "tell me about {company}", "what do you know about {company}"
# → agent_id="career-ops", skills=["research"], scoped_input={"company": <extracted name>}

# outreach — user asks to draft a message
# Pattern: "draft outreach", "message the hiring manager", "LinkedIn message for"
# → agent_id="career-ops", skills=["outreach"], scoped_input={"target": <context>, "context": <context>}

# triage — user has multiple postings to quick-score
# Pattern: "triage these", "quick score", "rank these postings"
# → agent_id="career-ops", skills=["triage"], scoped_input={"postings": <parsed list>}

# compare — user wants to compare options
# Pattern: "compare these", "which is better", "side by side"
# → agent_id="career-ops", skills=["compare"], scoped_input={"postings": <context>}

# track — user asks about applications
# Pattern: "show my applications", "application status", "what have I applied to", "track this"
# → agent_id="career-ops", skills=["track"], scoped_input={"action": <inferred>}

# apply — user wants to prepare an application
# Pattern: "help me apply", "fill out this application", "cover letter for"
# → agent_id="career-ops", skills=["apply"], scoped_input={"posting": <context>}
```

**Important:** These are NOT hardcoded regex patterns. They're guidance for the LLM-based intent routing that Roberto already uses. The review cycle (observation.py) and Roberto's react loop both have access to the registry's agent/skill descriptions. The patterns above document the *expected* triggering behavior so we can verify it works.

### Two Distinct Trigger Paths

**Path 1 — Telegram direct (immediate, user-initiated):**
Daniel sends "evaluate this posting" → Roberto's react loop recognizes the intent via registry skill descriptions → Roberto calls `spawn_subagent(agent_id="career-ops", skills=["evaluate"], trigger="telegram", ...)` → Block 2 registry resolves → Block 1 runtime executes → summary generated → result surfaces in Telegram.

**Path 2 — Review cycle (async, manager-initiated):**
The observation.py manager sees accumulated career data and decides a skill should run (e.g., "Daniel hasn't evaluated this interesting posting in the queue") → manager outputs a `subagent_spawns` entry → dispatch at observation.py line 1090 calls `spawn_subagent(trigger="review_cycle", ...)`.

**Telegram handler dependency (step-82, not step-83):**
`bregger_core.py` lines 2692-2727 are currently hardcoded to `test-echo` with an inline checklist. For Path 1 to work, the handler must use the registry to resolve agent + skills instead of inline construction. This is a step-82 deliverable (the handler needs the registry to exist). Step-83 validates that career-ops works through both paths but does not modify the Telegram handler itself.

The KeywordRouter has `subagent_spawn` regex patterns (lines 1566-1573) expecting `run subagent career-ops` syntax. Natural language like "evaluate this posting" routes through Roberto's react loop, not regex. Both paths converge at `spawn_subagent()`.
