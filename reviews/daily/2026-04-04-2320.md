## step-49 Spec Summary

**Title:** Thread Lifecycle Management
**Why:** 111 threads all permanently `active`. Nothing ever marks them `stale` or
`resolved`. The observation cycle queries `WHERE status = 'active'`, so it scans the
full history every tick — quality degrades as thread count grows.

**What was written:**
- `xibi/threads.py` — `sweep_stale_threads()` and `sweep_resolved_threads()` (pure SQL,
  synchronous, no LLM calls, idempotent)
- `xibi/heartbeat/poller.py` — `_sweep_thread_lifecycle()` called once per calendar day
  at the start of `async_tick()`, following the `_cleanup_telegram_cache()` pattern
- `xibi/command_layer.py` — `resolve_thread()` + `/resolve <thread_id>` Telegram command
- `tests/test_thread_lifecycle.py` — minimum 15 tests covering sweep correctness,
  idempotency, deadline handling, operator command, and heartbeat gate

**Staleness rules:**
- `active` → `stale` after 21 days no new signals (updated_at)
- `stale` → `resolved` after 45 days stale
- `active` → `resolved` if current_deadline passed 7+ days ago

**Scope:** additive-only. No schema migration. No async. Follows existing patterns
exactly.
