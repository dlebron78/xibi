## Notes

**CI fix (commit 224b667): `pop_digest_items` — BEGIN IMMEDIATE.**
The previous step-12 commit (260759a) introduced `pop_digest_items()` using
Python's default `with conn:` context manager, which issues a DEFERRED
transaction.  In SQLite WAL mode, deferred transactions allow concurrent
readers to both read the same rows before either commits — so two
simultaneously-running `HeartbeatPoller.digest_tick()` calls could process
the same digest items, producing duplicate alerts.

`test_watermark_race_condition_safe` caught this race, but the test passed on
Python 3.10 (local) due to GIL-serialized thread scheduling, and failed
deterministically on Python 3.11 in CI (different thread scheduler behavior
under coverage measurement).  Both CI attempts 1 and 2 failed.

Fix: changed `pop_digest_items` to use `isolation_level=None` (autocommit
mode) and issue `BEGIN IMMEDIATE` explicitly.  This acquires the write lock
before any reads, fully serializing concurrent callers.  534/534 tests pass;
CI run #23643099663 (Python 3.11) is green.
