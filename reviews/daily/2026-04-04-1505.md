necessary. The spec defines `SourcePoller` with `async def poll_due_sources()` and requires
`tick()` to `await` the poller. This creates an await chain that propagates up. Jules is
correct that MCP calls (subprocess I/O) cannot run synchronously in the heartbeat path. The
async refactor is justified. However, Opus must verify at merge time that:

- The refactor is scoped to the async path, not a wholesale rewrite
- Sync-only code paths (Telegram polling, ReAct CLI) degrade gracefully if awaited in a sync
  context
- No `asyncio.run()` nesting is introduced that could cause runtime errors

technical direction but carries risk: changes to `Executor`, `react.run`, and
`ObservationCycle` touch the entire ReAct loop. If Jules over-refactored (e.g. converted
every method to async without testing sync paths), the change could break the Telegram
polling loop or CLI mode. The Opus review must scrutinize this carefully. Flag for Opus:
check that sync callers of react.run/executor are not broken.
