## Issues Found and Resolution

### BLOCKING (fixed before merge)

**Leaked Jules AI scratch files** — `final_surgical_fix.py` and `fix_react_sync.py` were committed to the repo root. These are Julius AI code-editing helper scripts from the sync/async migration work that had no place in production code. Removed in commit `6ec061d` before merge.

### FALSE POSITIVES (Opus misreadings)

**nudge.py missing run() wrapper** — Opus flagged `xibi/skills/nudge.py` as missing a `run()` function. In fact, the executor calls `xibi/skills/sample/nudge/tools/nudge.py:run()`, which has a correct hybrid sync/async wrapper. `xibi/skills/nudge.py` is business logic only, not a tool file. No issue.

**Hybrid sync/async pattern coroutine return** — Opus flagged that `run()` returns a bare coroutine in async contexts. In practice, `_handle_text` in telegram.py is a sync function (no running loop), so `asyncio.run()` is always used there. All async callers in the heartbeat poller use `await`. The footgun exists architecturally but doesn't affect any current call site.

### NON-BLOCKING (follow-up items)

- **SourcePoller executor integration semantics**: The poller `await`s `self.executor.execute()` which is sync internally — works but is semantically odd. Consider making this explicit.
- **Signal extractor silent failures**: `extract()` returns empty list on error — operators may miss misconfigured extractors. Consider ERROR-level logging or metrics.
- **nudge.py config loading complexity**: Multi-path config loading (YAML, JSON, multiple env vars, .xibi_env) silently continues on all exceptions. Structured validation would improve debuggability.
- **Extractor name validation**: Config's `signal_extractor` field is not validated at load time — typos produce silent warnings. Validate at startup.
