### Vision notes

The implementation is architecturally sound in all respects except tier resolution. The MCP client design (synchronous blocking, select() for timeout, normalized result dicts, failure isolation) is exactly right for Xibi's single-GPU, synchronous model. The schema rename (parameters → inputSchema) is complete and correct, finally enabling schema validation that was silently skipped everywhere.

The blocking issue is one line: `DEFAULT_TIER = PermissionTier.GREEN` must become RED. This was the sole purpose of step-35a Fix 1. The PR ships MCP Foundation with a security regression: any MCP server the user configures can execute arbitrary tool calls without user confirmation, because the permission system resolves their tier to GREEN.

**Trade-off to flag for Daniel:** Setting DEFAULT_TIER = RED will affect all tools not in TOOL_TIERS — not just MCP tools. If there are any custom or ad-hoc tools that currently rely on auto-execution via DEFAULT_TIER = GREEN, they will break and require explicit TOOL_TIERS registration. This is the correct behavior (explicit > implicit for safety) but Daniel should be aware.

### Summary

PR #38 builds a solid MCP Foundation: the client, registry, executor, and schema rename are all well-implemented. CI is clean. 12 of 13 checklist items pass. The single hard failure — MCP tools getting GREEN tier instead of RED through command_layer's resolve_tier() path — is a direct contradiction of the spec's primary safety constraint and a real security regression. Until DEFAULT_TIER = RED (or equivalent) is implemented and tested, this cannot merge. The fix is one line plus three tests. Everything else is ready.
