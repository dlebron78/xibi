### Operational readiness
- ✅ DB writes use `open_db()` from `xibi/db/__init__.py` throughout (kernel.py lines 96, 134, 233; api.py all paths)
- ✅ Kernel tick runs synchronously inside the heartbeat; exceptions caught + logged with `exc_info=True`
- ✅ No long-running write transactions; parent action row updated once per action after handler completes
- ✅ Env vars / config validated at startup (db_path passed at heartbeat init, kernel gracefully skipped if executor None)
- ✅ Run row is the source of truth for "did this action start?" — written before dispatch, so SIGTERM mid-handler leaves a recoverable `status='running'` row
- ✅ Phase 1.5 is bounded by the heartbeat's _PHASE15_TIMEOUT_SECS=60 and per-action timeout
- ⚠️ **Timeout class minor gap**: `Timeout` in `kernel.py:39-63` uses `ctypes.pythonapi.PyThreadState_SetAsyncExc` with `c_long` for thread ID. Spec's TRR record approved a "fallback to skip if unavailable" design, but the code has no try/except around the async-exception raise. In practice, on CPython 3.7+ (target platforms) it works; worst case the exception is caught by the general `except (TimeoutError, Exception)` at kernel.py:189 and marked as an error instead of a graceful skip. Recommend a follow-up micro-commit wrapping the `PyThreadState_SetAsyncExc` call in `try/except (AttributeError, RuntimeError)` and logging a warning. **Not blocking** — TRR pre-approved this design.

### Vision notes
- Scheduling is now the single source of truth for periodic work in Xibi. Future follow-ups can migrate the hardcoded heartbeat hooks (manager review cadence, Jules audit, observation cycle time gate) to scheduled_actions with `internal_hook` handlers — this is the architectural unlock this step buys.
- Intentional deferrals (captured in spec and TRR, not regressions): no jitter in interval trigger, no catch-up semantics for missed slots, no auto-disable notification channel, no cron parser, no ReAct tool surface for user-facing registration.
- `trust_tier` is set at registration and never mutated — attack surface for tier escalation is closed by design.
- Phase 1.5 placement means scheduled actions can influence signal extraction in the same tick (e.g., refresh task list, adjust trust tiers) — a future spec can exploit this.
