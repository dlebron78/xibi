### What step-38 builds

**Belief Protection: Session Source Tagging** — a security-correctness fix that prevents
MCP tool responses from writing to long-term memory.

**Problem:** `compress_to_beliefs()` treats all session turns equally. The `source` field
already exists on the `Turn` dataclass and in the `session_turns` table (default: `"user"`),
but:
1. Nothing in the system currently passes a non-default source
2. `compress_to_beliefs()` does not filter by source

Since step-35 shipped MCP Foundation (MCPClient, MCPServerRegistry, MCPExecutor), an MCP
tool can now execute during a ReAct loop. If that tool response contains text like "user
confirmed X", that text ends up in the ReAct answer, which then gets compressed into
persistent beliefs — surviving future sessions without the user having explicitly stated it.

**Fix (two-part, contained):**
1. `compress_to_beliefs()` adds `AND source = 'user'` to the SQL query — only user-initiated
   turns can write to long-term memory.
2. `TelegramAdapter._handle_text()` detects MCP tool usage in the `ReActResult` via
   a new helper `_detect_mcp_source()` that checks `step.tool` against the skill registry,
   then passes `source="mcp:{server_name}"` to `add_turn()`.

This is the exact fix described in the BACKLOG MCP Roadmap section under:
> [P2] Belief protection — session source tagging
> Depends on: step-35. Prerequisite for: any MCP server that returns free-text responses.
