## Root causes fixed

**1. `_migration_7` placed outside `SchemaManager` class (critical)**
In `xibi/db/migrations.py`, `_migration_7` was accidentally written after the
module-level `migrate()` function rather than inside the class body. Python silently
treated it as dead code. Result: `SchemaManager` had no `_migration_7` attribute,
and every test that called `schema.migrate()` (30+ tests) crashed with `AttributeError`.

Fix: moved `_migration_7` inside the class, also replaced bare `try/except/pass`
with `contextlib.suppress(sqlite3.OperationalError)` (SIM105 lint fix).

**2. Missing `_with_conn` variants on `RuleEngine` (critical)**
`xibi/heartbeat/poller.py` calls `rules.get_seen_ids_with_conn(conn)`,
`rules.load_triage_rules_with_conn(conn)`, `rules.log_signal_with_conn(...)`,
`rules.log_triage_with_conn(...)`, `rules.mark_seen_with_conn(...)` — but none of
these methods existed on `RuleEngine`. Added all 7 `_with_conn` variants that accept
a shared connection, enabling the tick loop to be fully atomic.

**3. `digest_tick` used wrong atomicity pattern (correctness)**
After the refactor, `digest_tick` was calling `rules.get_digest_items_with_conn(conn)`
inside a shared-connection transaction — but this breaks the purpose of `pop_digest_items`,
which uses `BEGIN IMMEDIATE` for its own locking to prevent duplicate digest sends.
Two tests (`test_watermark_race_condition_safe`, `test_digest_tick_uses_pop_digest_items`)
were failing. Reverted `digest_tick` to use `rules.pop_digest_items()` (the correct
atomic method). Removed the outer DB transaction from `digest_tick` — it was redundant
given `pop_digest_items` manages its own locking.

**4. Dashboard `get_db_conn()` return type (mypy)**
Fixed return type from `Connection` to `AbstractContextManager[Connection]` to match
the `@contextmanager`-decorated `open_db()` return.

**5. Test suite updated**
Added `_make_mock_db_ctx()` helper in `test_poller.py` that yields an in-memory SQLite
connection seeded with all required tables. Updated 8 tick tests to patch `xibi.db.open_db`
with this context manager and updated assertions to check `_with_conn` method names.
Seeded `heartbeat_state` in `test_tick_task_reminder` and `test_tick_no_tasks_table`.

## Architecture notes

The `_with_conn` pattern introduced in step-16 is the right direction — sharing a
single connection across all tick operations prevents partial writes if the poller
crashes mid-tick. However, `digest_tick` should remain on `pop_digest_items` because
that method uses `BEGIN IMMEDIATE` (write-lock) to prevent two concurrent callers
from reading the same triage items. Mixing both patterns in the same tick loop was
the source of confusion. The invariant going forward:

- `tick()` → shared connection (`_with_conn` variants), 1 atomic transaction per tick
- `digest_tick()` → `pop_digest_items()`, which handles its own transaction locking
