## Step-33 spec notes
Trust gradient integration wires the existing `TrustGradient` class (fully implemented in step-11)
into the three execution paths that produce model outputs:
1. `enrich_signals()` — fast role batch extraction → trust.record_success/failure("text","fast")
2. `ObservationCycle._run_role_loop()` — review/think roles → schema failures update trust
3. `Radiant.run_audit()` → quality degradation updates "text.review" trust

Key design decisions in the spec:
- `should_audit()` gates tier-1 extraction in enrich_signals (probabilistic skip on low-trust batches)
- Trust recording is best-effort everywhere — no trust failure affects primary operation
- Lazy imports in poller.py to avoid circular dependencies
- No new DB migrations needed (trust_records already exists from migration 4)
- summary() trust key reads raw SQL, not via TrustGradient instance (keeps radiant self-contained)
