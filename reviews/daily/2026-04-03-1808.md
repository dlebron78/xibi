### Vision notes
1. **Category filtering behavioral change (minor):** `recall.py` now filters category post-query for ledger rows (Python loop) instead of in the SQL WHERE clause. If >15 ledger rows exist and most are wrong category, some results could be missed at the LIMIT boundary. Low practical impact now. Worth adding category to the SQL WHERE when time permits.
2. **Beliefs have no category field:** `recall.py` doesn't filter beliefs by `category` param — beliefs only have `type`. This asymmetry is probably intentional (beliefs come from session compression, not user-categorized memory) but is undocumented.
3. **Error message variance:** Without the explicit `db_path.exists()` guard, error message quality depends on sqlite3's errno. The test's 3-way assert (`"Database not found" OR "No such file or directory" OR "unable to open database file"`) captures this adequately.
4. **manage_goal.py old fallback removed:** Old code fell back to `project_root` (the tool's own parent directory) as the workdir default — that was effectively a silent misconfiguration. New code falls back to `~/.xibi` which is the correct default. This is a behavioral improvement.

### Summary
This PR fixes a critical production regression that has made the entire memory system write-only from the model's perspective: all five tools were building paths to a `bregger.db` file that no longer exists after the Bregger→Xibi migration. The implementation is correct and complete — `open_db()` is used consistently, dual-table recall is properly implemented with source tagging and exclusion of expired and marker beliefs, and the eight required integration tests verify the critical behaviors against a real database. No architectural violations. The minor notes are cosmetic or low-risk behavioral details that don't affect correctness.
