## CI Fix

Before the Opus review, CI had a typecheck failure:

```
xibi/react.py:255: error: Incompatible types in assignment
  (expression has type "LLMRoutingDecision | None", variable has type "RoutingDecision")
xibi/react.py:257: error: "RoutingDecision" has no attribute "skill"
... (8 total errors)
```

Root cause: Jules reused the name `decision` for the LLM classifier result (line 255), but `decision` was already typed as `RoutingDecision` earlier in the function (from `control_plane.match()`). mypy correctly flagged the type collision.

Fix: Renamed the local variable to `llm_decision` in the LLM classifier block. 1-line change, no logic impact. Committed and pushed to the PR branch. CI re-ran and passed (all 4 checks green).

### Vision notes

This step brings meaningful quality improvement for deployed Xibi. Queries that use non-canonical phrasings (paraphrases, novel vocabulary) previously fell through to pure ReAct without any routing hint. The LLM classifier now provides a skill+tool suggestion as context, improving tool selection accuracy on the first ReAct step. Cost impact is low: classifier only fires on BM25 miss, uses a `fast` model, and is wrapped so failures are silent.

One trade-off to note: `LLMRoutingClassifier` is always instantiated in `cmd_telegram()` with the current code. If the config has no `text.fast` model (unlikely in production but possible in minimal test setups), the classifier will fail silently on first use. This is acceptable per spec (try/except) but Daniel should be aware that if classifier errors appear in logs, it likely means the `fast` model is misconfigured rather than a code bug.

The CI fix (variable rename) was a trivial mypy type collision with no behavioral impact. Architecture boundaries are respected. Ready to ship.
