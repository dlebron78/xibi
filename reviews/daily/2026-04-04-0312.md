## Opus Review — PR #49 (step-46)

**Verdict: REJECT**

### Spec alignment

✅ All 7 files in spec's file structure present in diff

✅ No LLM calls in resolution path — pure Python + SQL

❌ Constraint violated — spec requires "all DB writes use open_db()" — violated in send_email.py (see below)
✅ Function signatures match spec — resolve_contact(), bridge_to_contacts(), add_chitchat_turn() all match

### Architecture alignment

✅ No model name references — resolver is pure Python+SQL
✅ Channel/tool/skill terminology correct — channel_type values ("email", "session", "slack") consistent with architecture
✅ No asyncio introduced
✅ No MCP involved (not applicable)
✅ Schema migration pattern respected — migration_18() added to migrations.py properly
⚠️ source tagging — session_entities gets contact_id via bridge, but the source column semantics are not touched (bridge doesn't add a new source type). Acceptable — bridge is additive enrichment, not a new source.

### Test quality

✅ Tests assert meaningful behavior (not just "runs without error")
✅ 14 tests meet the spec requirement
✅ No real model calls in tests
❌ Failure-path coverage thin — only 1 of 6 resolver tests covers a failure case (ambiguous match). Tests for invalid db_path, UNIQUE constraint violations in contact_channels, and exception propagation from bridge_to_contacts are absent.
✅ Happy paths covered across all 3 test modules

### Operational readiness

❌ **HARD BLOCK: `_track_outbound()` in skills/email/tools/send_email.py uses bare `sqlite3.connect(db_path)` at lines 99 and 116.** This bypasses WAL mode and busy timeout configuration that open_db() provides. On NucBox where heartbeat and email sender share the DB, this risks write contention and journal corruption. Fix: replace with `with open_db(Path(db_path)) as conn, conn:`.

❌ **HARD BLOCK: `bridge_to_contacts()` has no exception handling.** The method is described in the spec as "additive, not required" — but if resolve_contact() or create_contact() raises (schema not migrated, locked file, missing table), the exception propagates out of `add_turn()` and can crash the active session. The chitchat fast-path (step-45) correctly uses try/except with fallthrough. bridge_to_contacts() must do the same.

✅ `with open_db(...) as conn, conn:` pattern in session.py is correct — this is an established pattern (used in 4 existing locations in session.py). `open_db()` yields a `sqlite3.Connection`; the second `conn` starts an explicit transaction via `sqlite3.Connection.__enter__`. Not a bug.

⚠️ Migration 18's `except sqlite3.OperationalError: pass` block is too broad — it silences column-already-exists errors (intentional) but also disk-full and permission errors (not intentional). Low severity but should be tightened.

⚠️ `handle LIKE '%@' || domain` forces a full table scan (leading wildcard defeats index). Acceptable for current scale, flag for future.

### Vision notes

This step moves Xibi meaningfully toward a better personal agent. The centralized contact model is the right foundation — when the owner replies to an email, `user_endorsed=1` is set, and that signal propagates to tier resolution (step-44) and future lead nurturing. The architecture (single contacts table + contact_channels for multi-channel identity) is clean and will scale.

Two things Daniel should know before it ships:
1. The bare sqlite3.connect() in send_email.py is the kind of oversight that's easy to miss in a large diff — it's a pattern Jules used in the skills layer (which has historically been looser) that conflicts with the xibi/ constraint. Worth flagging to Jules as a constraint reminder.
2. bridge_to_contacts() is the right design but needs the safety net — the entire session shouldn't fall over because we couldn't link a contact.

### Summary

Step-46 builds the right data layer. The contact resolution logic is clean, the migration pattern is correct, the email enrichment and outbound tracking are well-structured, and the context block enrichment is a nice UX improvement. Two implementation gaps block merge: bare sqlite3.connect() in the skills layer (bypasses WAL settings) and missing try/except in bridge_to_contacts() (violates the "additive, not required" design intent). Both are small, targeted fixes. Once Jules addresses them, this PR should clear Opus review on next run.
