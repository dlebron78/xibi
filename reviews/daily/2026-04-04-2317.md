### Vision notes

- The multi-source framework design is excellent and exactly what Xibi needs. Config-driven source registration means adding a source is a config entry, not a code change — this is the right abstraction.
- The async tick conversion is reasonable — the source poller being async pulls tick async. The bug is specifically `await sync_function()` which in Python silently works for coroutines but raises TypeError for plain sync returns. If executor.execute() returns a dict directly, `await dict` → TypeError.
- Nudge now properly delegates: sample/nudge/tools/nudge.py → xibi/skills/nudge.py. Clean layering.
- The observation cycle and react loop now use the async pattern (check running loop → return coroutine or run). This is a pragmatic pattern for gradual async adoption.

### Summary

PR #52 implements the complete multi-source framework per spec: SourcePoller, SignalExtractorRegistry, nudge skill, MCP resources, and observation cycle resource injection are all present and architecturally correct. The design is clean and config-driven. However, the PR has two critical runtime bugs — source_poller._poll_source() awaits executor.execute() and client.call_tool(), both synchronous, causing TypeError crashes on the first tick. Additionally, 6 of 21 required tests are absent (MCP resources tests, integration tests, source poller failure paths). The code cannot run in production without the async/await fix, and the missing tests leave key behaviors unvalidated. REJECTED pending fixes to these specific issues.
