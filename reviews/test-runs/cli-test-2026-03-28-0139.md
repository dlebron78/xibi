## Failure Pattern Analysis

### Pattern 1: Tool name vs. skill name mismatch (CRITICAL — all suites)
The most pervasive failure. The LLM calls `tool="email"` but the executor dispatches to a function named `email` inside `handler.py`. No such function exists — the available functions are `list_emails` and `triage_email`. This produces `Unknown tool: email` on virtually every email-related turn.

**Root cause:** phi4-mini conflates the skill name (from manifest's `"name"` field) with the tool name (from `"tools[].name"`). The system prompt presents the full manifest, but phi4-mini learns "email is the tool" rather than "email is the skill containing tools list_emails and triage_email."

**Fix:** Either (a) flatten the tool list in the system prompt so tools are presented as `list_emails`, `triage_email` etc. directly, or (b) add a skill-name-to-tool resolution shim in the executor that picks the first tool when the skill name is called with no args.

### Pattern 2: Test runner I/O capture desync (CRITICAL — all suites)
The subprocess I/O reader uses `select()` with 0.1s polling and a 30s deadline. phi4-mini takes 7–22 seconds per turn. The result: turn N+1's response reader picks up leftover output from turn N's processing (the prompt's `→ step` lines, error text, etc.). Turn 1 of every suite captures the CLI startup banner instead of the actual response to the first query.

**Fix:** The runner needs to wait for a recognized end-of-turn marker (e.g., `xibi> ` on a new line) rather than a quiet period.

### Pattern 3: Shadow matcher misfiring (2 instances)
- Suite 5 Turn 3: "schedule a reminder to check on that in 3 days" → shadow matched `web_search` (direct), completely bypassing the ReAct loop and ignoring user intent
- Suite 2 Turn 1: shadow-hint activated `list_events` but the runner didn't wait for the actual ReAct result

**Fix:** Shadow phrases for `web_search` likely match too broadly. Review shadow phrase corpus for `web_search` to avoid capturing schedule-adjacent queries.

### Pattern 4: Context not injecting into subsequent turns
Since tool calls all fail (Pattern 1), `session.add_turn()` records turns with empty `tools_called` and no useful answer. `get_context_block()` returns references to failed turns. The "context not injecting" symptom is downstream of Pattern 1 — fix tool routing and context will populate.

### Pattern 5: Entity extraction JSON parse errors
Background warning: `Entity extraction failed: Extra data: line N column M`. phi4-mini returns JSON with trailing explanation text or wraps output in markdown fences. `generate_structured()` expects raw JSON.

**Fix:** Strip markdown fences before `json.loads()` in `generate_structured()`, and/or use `format="json"` at the top level of the Ollama API call (currently it's placed inside `options` dict where Ollama ignores it).
